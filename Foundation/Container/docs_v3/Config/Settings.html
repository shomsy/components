Settings.php

A lightweight configuration storage class that provides dot-notated key access to configuration arrays.

File Path: Config/Settings.php
Folder Context: Configuration management
Updated: 2026-01-09

Quick Summary

Settings.php implements a simple key-value configuration store with dot notation support. It allows storing and retrieving configuration values using hierarchical keys like "database.host", making it easy to organize settings without complex structures.

This file exists to provide a self-contained configuration storage mechanism that doesn't require external dependencies, focusing specifically on container operational settings rather than application business data.

For Humans: What This Means

This is like a smart notebook for the container's settings. You can write down preferences using simple dotted paths, and later ask for them back. It keeps things organized without needing a full database or complex setup.

Terminology

  • Dot Notation: Using periods in keys (e.g., "app.cache.enabled") to create nested configuration hierarchies
  • Configuration Store: In-memory storage for settings that can be accessed by key
  • Default Values: Fallback values returned when a requested key doesn't exist
  • Hierarchical Keys: Keys that represent nested structure through dot separation

Story Example

You're configuring the container for different environments. In development, you want debug logging and fast but unsafe caching. In production, you want minimal logging and slow but reliable caching.

The Problem

Without a configuration system, you'd have conditional logic scattered throughout the codebase checking environment variables directly.

The Solution

Settings.php provides a clean abstraction where you can load environment-specific configurations once, then access them consistently throughout the container.

For Humans: What This Means

This solves the problem of "how do I make my container adapt to different environments without changing code?" You load the right settings at startup, and the container behaves appropriately without knowing which environment it's in.

For Dummies

Settings.php is like a key-value store where keys can have dots to make them organized.

Simple Example

$settings = new Settings();

// Store some values
$settings->set('database.host', 'localhost');
$settings->set('database.port', 3306);
$settings->set('debug', true);

// Get them back
$host = $settings->get('database.host'); // 'localhost'
$port = $settings->get('database.port'); // 3306
$debug = $settings->get('missing', false); // false (default)

Why Dots?

Dots let you group related settings: all database stuff under "database.*", all cache stuff under "cache.*".

For Humans: What This Means

This is the container's way of remembering things. You tell it to remember "database.host" is "localhost", and later you can ask "what's the database host?" and it tells you. It's simple but powerful for keeping configuration organized.

How It Works

The class uses PHP arrays internally with string manipulation for dot notation:

  1. Storage: Single array holds all configuration data
  2. Get Operation: Splits key on dots, navigates array structure
  3. Set Operation: Creates nested arrays as needed, assigns final value
  4. Has Operation: Traverses structure to check existence

For Humans: What This Means

Under the hood, it's just PHP arrays with some smart string splitting. When you say "database.host", it splits on the dot and looks in $array['database']['host']. Setting creates the nested structure automatically.

Architecture Role

This class serves as the foundational configuration storage layer in the container's architecture. It provides the basic CRUD operations for settings that higher-level components build upon.

What This Depends On

For Humans: What This Means

This is the bottom layer of the configuration pyramid. Everything else that needs settings eventually calls down to this class. It's deliberately simple so that complex configuration logic can be built on top.

Risks, Trade-offs & Recommended Practices

Memory Usage

Risk: All configuration kept in memory simultaneously.

Design Stance: Optimized for container startup performance over memory efficiency.

Recommended Practice: Keep configurations reasonably sized for container operational settings.

For Humans: What This Means

Everything stays in memory, so don't load your entire application's configuration here. It's meant for the container's settings, not user data or massive datasets.

Type Safety

Risk: Mixed types in configuration without validation.

Design Stance: Flexibility prioritized over strict typing for configuration scenarios.

Recommended Practice: Validate configuration at application boundaries.

For Humans: What This Means

You can store any type of value (strings, numbers, arrays), which is convenient but means you need to be careful about what you put in and how you use it later.
