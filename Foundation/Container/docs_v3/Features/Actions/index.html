Actions Folder

Executable operations and service manipulation capabilities that the container can perform, including injection, instantiation, invocation, and resolution actions.

Folder Path: Features/Actions/
Context: Container action implementations
Updated: 2026-01-09

Quick Summary

The Actions folder contains the concrete implementations of executable operations that the container can perform on services. It provides the mechanisms for dependency injection, service instantiation, method invocation, and advanced resolution strategies.

This folder exists because containers don't just store services—they need to actively manipulate and prepare them. Actions are the verbs that make the container do things, not just hold things.

For Humans: What This Means

This is where the container becomes active rather than passive. Instead of just being a storage system, it can actually do work: inject dependencies, create objects, call methods, and resolve complex scenarios. It's the container's toolkit for getting things done.

Terminology

  • Inject: The process of providing dependencies to objects
  • Instantiate: Creating new object instances
  • Invoke: Calling methods or functions with dependency injection
  • Resolve: Finding and preparing services for use
  • Advanced: Sophisticated action implementations for complex scenarios
  • Action Pipeline: Sequence of operations performed during service preparation

For Humans: What This Means

Inject puts things into objects. Instantiate makes new objects. Invoke calls methods. Resolve finds and prepares services. Advanced handles tricky cases. The pipeline is the step-by-step process that turns raw services into usable components.

Story Example

You're building a content management system where articles need to be created with their dependencies (database, cache, logger), rendered with proper formatting, and saved with metadata tracking.

The Problem

Manual object creation and dependency management leads to scattered code and inconsistent initialization patterns.

The Solution

The Actions folder provides standardized mechanisms for instantiation, injection, and invocation that ensure consistent service preparation across the application.

For Humans: What This Means

This solves the problem of "how do I consistently create and prepare complex objects?" Instead of writing custom initialization code everywhere, you get standardized actions that handle the common patterns automatically.

For Dummies

The Actions folder is like the container's action verbs. It contains the code that actually does things with services, rather than just storing them.

What Actions Can Do

The container can:

  • Inject: Put dependencies into objects
  • Instantiate: Create new objects from classes
  • Invoke: Call methods with automatic dependency injection
  • Resolve: Figure out how to provide requested services
  • Advanced: Handle complex scenarios and edge cases

For Humans: What This Means

This is what makes dependency injection useful. It's not just about storing objects—it's about actively managing how they're created, connected, and used. Without actions, the container would be like a warehouse with no forklifts.

How It Works

Actions operate through coordinated pipelines:

  1. Trigger: Action requested by kernel or application code
  2. Analysis: Determine what needs to be done based on context
  3. Execution: Perform the specific action (inject, instantiate, etc.)
  4. Integration: Coordinate with other actions in the pipeline
  5. Completion: Return prepared service or confirmation

For Humans: What This Means

Actions work like a well-choreographed dance. Each action knows its role and timing, and they coordinate to produce the final result. The container orchestrates this automatically based on what you ask for.

Risks, Trade-offs & Recommended Practices

Performance Overhead

Risk: Complex actions can slow down service resolution.

Design Stance: Actions are optimized and cached where possible.

Recommended Practice: Use simple actions for performance-critical paths.

For Humans: What This Means

Actions are powerful but not free. For high-performance scenarios, you might want to use simpler approaches. The trade-off is between convenience and speed.

Complexity Management

Risk: Advanced actions can be difficult to understand and debug.

Design Stance: Actions are modular and well-documented.

Recommended Practice: Start with simple actions, add complexity as needed.

For Humans: What This Means

Don't use advanced actions just because they're cool. Keep it simple unless you actually need the complexity. Simple actions are easier to debug and maintain.
