ResolutionPipeline

Orchestrates the sequence of resolution steps for a service.

Foundation/Container/Core/Kernel/ResolutionPipeline.php

Class Overview

ResolutionPipeline implements the Pipeline pattern to execute a series of atomic steps that transform a request into a resolved service instance.

More

The pipeline is the heart of the container's runtime logic. Instead of a monolithic "resolve" method, the process is broken down into small, single-purpose steps (e.g., check scope, resolve bindings, inject dependencies).

This design allows for:

  • Modularity: Steps can be added, removed, or reordered without changing the core engine.
  • Observability: The pipeline emits events (Start, Success, Fail) for every step, enabling detailed telemetry.
  • Atomicity: If any step fails, the entire resolution creates a clear exception trail, making debugging easier.

The pipeline passes a KernelContext object through each step. Steps modify this context or use it to look up information.

Story Example

Constructing and running a simple pipeline.

use Avax\Container\Core\Kernel\ResolutionPipeline; use Avax\Container\Core\Kernel\Contracts\KernelContext; // 1. Define steps $steps = [ new CheckCacheStep(), new BuildInstanceStep(), new SaveToCacheStep() ]; // 2. Create pipeline $pipeline = new ResolutionPipeline($steps, $telemetry); // 3. Run pipeline $context = new KernelContext('UserService'); $pipeline->run($context); $instance = $context->resolvedInstance;
More

In this story, we see the pipeline purely as a runner. It doesn't know what "CheckCacheStep" does; it just executes it.

If "CheckCacheStep" finds an instance, it might mark the context as resolved. If the pipeline encounters a TerminalKernelStep and the context is resolved, it stops early (short-circuiting). This is how caching makes resolution fast.

Methods

run

Execute the pipeline steps in order.

More

Iterates through the registered steps. For each step:

  1. Records start time.
  2. Emits `StepStarted` event.
  3. Executes the step callable.
  4. Emits `StepSucceeded` event.
  5. Checks if it should stop (if terminal step + resolved).

If an exception occurs, it emits `StepFailed`, wraps the exception with pipeline context (step index, class name), and re-throws it.

withStep

Return a new pipeline with an appended step.

More

The pipeline is immutable. This method returns a new instance with the added step, preserving the existing specific configuration.
