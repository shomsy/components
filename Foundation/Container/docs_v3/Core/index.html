Core Folder

Fundamental container infrastructure providing the central runtime orchestration and configuration management for dependency resolution.

Folder Path: Core/
Context: Container runtime and orchestration
Updated: 2026-01-09

Quick Summary

The Core folder contains the essential runtime components that orchestrate all container operations. It implements the kernel pattern where ContainerKernel acts as the central coordinator, managing resolution pipelines, scoping, and injection through specialized subsystems.

This folder exists because dependency injection requires sophisticated coordination between multiple concerns—binding resolution, lifecycle management, and injection—all of which must work together seamlessly.

For Humans: What This Means

This is the container's brain and nervous system. While other parts handle specific jobs like configuration or security, this folder makes sure everything works together smoothly, like a conductor leading an orchestra where each musician plays their part perfectly.

Terminology

  • Kernel: The central orchestrator that coordinates all container operations
  • Resolution Pipeline: A sequence of processing steps that transform service requests into instances
  • Runtime: The execution environment where services are instantiated and managed
  • Facade: A unified interface that simplifies access to complex subsystems
  • Context: Metadata that accompanies resolution requests to guide the process
  • Orchestration: The coordination of multiple components to achieve a unified goal

For Humans: What This Means

These terms describe how the core acts as a smart manager. The kernel is the boss who delegates work, pipelines are the assembly lines, runtime is the workshop, and facades are the front desks that make complex systems easy to use.

Story Example

You're building a complex web application where user requests trigger a cascade of operations: authentication, database queries, business logic, caching, and response formatting. Each operation depends on multiple services with different lifecycles.

The Problem

Without centralized orchestration, each component would need to know how to create and manage all its dependencies, leading to scattered, hard-to-maintain code.

The Solution

The Core folder provides the orchestration layer where ContainerKernel manages the entire resolution process, ensuring each service gets its dependencies at the right time with proper lifecycle management.

For Humans: What This Means

This solves the problem of "how do I make a complex system where everything works together without everything knowing about everything else?" The core acts as the traffic controller that ensures each car (service) gets to its destination without collisions.

For Dummies

The Core folder is like the engine room of a ship. While the passengers enjoy the cruise and the crew handles specific tasks, the engine room makes sure everything runs smoothly.

Why Is This Important?

Dependency injection is complex because objects need other objects to work. This folder handles all the complexity of figuring out what needs what and when, so you don't have to.

Simple Mental Model

  1. You ask for a service
  2. The kernel checks if it exists or can be made
  3. The pipeline figures out what dependencies are needed
  4. Everything gets built in the right order
  5. You get a fully working service

For Humans: What This Means

This is the magic that makes dependency injection work. Instead of you worrying about the thousands of connections between objects, the core handles it all automatically, like a smart assistant who anticipates what you need before you ask.

How It Works

The core operates through a layered architecture:

  1. Entry Point: ContainerKernel receives service requests
  2. Fast Path: Check scopes for existing instances
  3. Resolution: Delegate to runtime through pipeline
  4. Instantiation: Build dependencies recursively
  5. Lifecycle: Apply scope rules and post-processing
  6. Return: Provide fully configured service instance

For Humans: What This Means

The core is like a professional kitchen brigade. When you order a dish, the chef (kernel) coordinates with the saucier (runtime), garde manger (scopes), and others to prepare each component perfectly, then presents you with a complete meal.

Architecture Role

This folder establishes the architectural core of the entire dependency injection system. It implements the composition root pattern and provides the stable runtime foundation that all other features build upon.

What Depends on This

  • All service resolution and instantiation
  • Scope and lifecycle management
  • Advanced features like injection and invocation
  • Monitoring and diagnostics systems
  • Security and policy enforcement

What This Depends On

  • PHP reflection system
  • Definition storage (bindings)
  • Configuration management
  • Basic language constructs

For Humans: What This Means

This is the foundation that everything else rests on. Like the engine in a car, it's not flashy, but without it, nothing moves. Every feature you use ultimately goes through this core system.

Risks, Trade-offs & Recommended Practices

Complexity Management

Risk: Core orchestration logic becomes difficult to maintain as features grow.

Design Stance: Modular architecture with clear separation of concerns.

Recommended Practice: Keep core logic focused, delegate specialized work to feature modules.

For Humans: What This Means

The core could become a tangled mess if everything is crammed here. Instead, it's designed like a well-organized control center where each button controls one specific system.

Performance Overhead

Risk: Multiple layers of indirection slow down service resolution.

Design Stance: Optimized fast paths and caching to minimize overhead.

Recommended Practice: Use singleton scopes for expensive services, enable compilation for production.

For Humans: What This Means

While the core does more work than direct instantiation, the performance cost is usually negligible compared to the benefits of maintainable, testable code.
