Container

Main facade for the dependency injection container.

Foundation/Container/Container.php

Class Overview

Container is the main public API for dependency injection. It provides a clean facade over the complex internal machinery, delegating all operations to the ContainerKernel.

More

The Container class exists to provide a stable, simple API that applications can depend on. It hides the complexity of resolution engines, pipelines, scopes, and prototypes behind a thin wrapper.

In the container lifecycle, the Container is instantiated early during bootstrap. It remains the primary interface for registering services and resolving dependencies throughout the application's runtime.

Terminology:

Abstract: the identifier used to request a service (often an interface).

Concrete: the class or factory that creates the instance.

Binding: the rule that maps abstract to concrete.

Scope: the lifetime boundary for shared instances.

Resolution: turning an abstract into a concrete instance.

Prototype: a precomputed build recipe used to avoid repeated reflection.

Pipeline: a fixed sequence of runtime steps in resolution.

Story Example

interface UserRepositoryInterface {
    public function findById(int $id): User;
}

class DatabaseUserRepository implements UserRepositoryInterface {
    public function __construct(private DatabaseConnection $db) {}
}

class UserService {
    public function __construct(private UserRepositoryInterface $repo) {}
}

$container = ContainerBuilder::create()->build();
$container->bind(UserRepositoryInterface::class, DatabaseUserRepository::class);
$service = $container->get(UserService::class);
More

This demonstrates dependency inversion. The UserService depends on an interface, not a concrete class. The container binds the interface to a concrete implementation.

Internally:

1) The binding is stored in the DefinitionStore.

2) ResolutionEngine inspects UserService constructor parameters.

3) It finds the binding for UserRepositoryInterface.

4) It builds DatabaseUserRepository, then UserService.

5) It caches instances if the lifetime requires it.

Methods

get

Resolve a service by identifier.

More

The primary resolution method. Checks scopes first for performance, then delegates to the kernel for full resolution pipeline.

WHY IT EXISTS:

- To provide the main entry point for service location (PSR-11)

- To enable dependency inversion by abstracting service creation

- To support different resolution strategies (bindings, autowiring, scopes)

- To cache resolved instances for performance

RESOLUTION FLOW:

1. Check active scopes for existing instances (fast O(1) lookup)

2. Check explicit bindings in DefinitionStore

3. Fall back to autowiring for concrete classes

4. Resolve constructor dependencies recursively

5. Inject property/method dependencies if configured

6. Cache instance in appropriate scope

SCOPE BEHAVIOR:

- Singleton: Instance cached globally, returned for all requests

- Scoped: Instance cached within current scope boundaries

- Transient: New instance created each time

ERROR SCENARIOS:

- ServiceNotFoundException: No binding or class found

- ResolutionException: Circular dependency or construction failure

PERFORMANCE CHARACTERISTICS:

- First resolution: Expensive (reflection, analysis, construction)

- Subsequent resolutions: Fast (scope lookup or cache hit)

- Memory usage: Scales with number of cached instances

has

Check if a service is registered or can be resolved.

More

Performs lightweight validation to determine if a service identifier can be resolved without actually constructing the service instance.

WHY IT EXISTS:

- To enable conditional service usage without triggering construction

- To support PSR-11 has() method for container interoperability

- To provide fast existence checks for application logic

- To avoid expensive resolution operations when not needed

CHECK ORDER:

1. Check active scopes for cached instances (fast O(1))

2. Check DefinitionStore for explicit bindings

3. Check if identifier is an instantiable class (reflection)

4. Return false if none of the above succeed

WHAT IT DOESN'T DO:

- Doesn't validate constructor dependencies

- Doesn't check if injection will succeed

- Doesn't perform any construction or caching

- Doesn't trigger lazy loading or initialization

USE CASES:

- Feature toggling based on service availability

- Optional dependency injection

- Plugin/extension systems

- Testing with mock service substitution

PERFORMANCE:

- Very fast: O(1) scope lookup + hash table checks

- No reflection beyond basic class_exists() calls

- Safe to call frequently in application code

make

Create a new instance with optional parameters.

More

Always creates a new instance, bypassing scope caches. Useful for factories or when you need multiple instances.

Parameters override automatic resolution, enabling test doubles and contextual construction.

Lifetime: always transient, never cached.

call

Call a callable with dependency injection.

More

Resolves callable parameters automatically, then executes it. Supports both functions and methods.

Enables dependency injection for code that doesn't use constructors (controllers, event handlers, etc.).

Parameters can override automatic resolution for testing or customization.

injectInto

Inject dependencies into an existing object.

More

Analyzes the target object for injection points (properties/methods) and resolves dependencies into them.

Useful for objects created outside the container (legacy code, factories, deserialization).

Respects scope rules - won't inject scoped instances outside their lifetime.

resolve

Resolve a service from a prototype.

More

Uses a precomputed prototype to build the instance. Bypasses analysis phase for performance.

Advanced API for when you have prototypes from compilation or analysis.

Still respects scope and binding rules.

resolveContext

Resolve a service with context.

More

Provides additional resolution context for advanced scenarios. Context can influence binding selection.

Used internally by the resolution pipeline for contextual dependencies.

Context objects contain metadata about the resolution request.

instance

Register a shared instance.

More

Stores an existing instance as a singleton. Future resolutions return this same instance.

Useful for registering external objects or pre-configured services.

The instance is stored in the singleton scope.

beginScope

Start a new resolution scope.

More

Creates a scope boundary for scoped services. Scoped instances live within this boundary.

Scopes are typically per-request or per-unit-of-work.

Multiple scopes can be nested; resolution checks parent scopes if not found locally.

endScope

End the current resolution scope.

More

Disposes all scoped instances and removes the scope boundary.

Important for cleanup and preventing memory leaks in long-running applications.

Triggers disposal callbacks if services implement them.

canInject

Check if an object can receive dependency injection.

More

Analyzes the object for injection attributes or compatible properties/methods.

Quick check without performing actual injection.

Useful for validation and tooling.

inspectInjection

Inspect injection points on an object.

More

Returns detailed information about what would be injected: properties and methods.

Includes resolved types for each injection point.

Used by tooling and for debugging injection issues.

scopes

Get the scope manager.

More

Provides access to the scope management system for advanced control.

Allows manual scope manipulation and inspection.

Use with caution - most applications don't need this.

exportMetrics

Export metrics as string.

More

Exports performance and usage metrics in string format.

Includes resolution counts, timings, cache hits, etc.

Useful for monitoring and performance analysis.

getDefinitions

Get the definition store.

More

Provides access to the internal definition storage.

Advanced API for tooling and bootstrappers.

Most application code should not need this directly.
