ContainerEngine

Runtime core that orchestrates resolve, scope, inject, and invoke flows.

Foundation/Container/Core/ContainerEngine.php

Class Overview

ContainerEngine is the execution core. It wires resolve pipeline, definition store, scope manager, prototype factory, and runtime utilities into a predictable workflow.

More

If the facade is the front desk, the engine is the workshop behind it. This is where the container decides which rule applies, builds objects, injects dependencies, and manages lifetimes.

Core concepts:

Definition: a stored rule for how to build a service (for example, interface to class mapping or a factory callback).

Prototype: a precomputed build plan created during the Think phase to reduce reflection cost at runtime.

Scope: a lifetime boundary for shared instances (singleton or per-request).

The engine keeps these concerns separate so you can change how services are built without changing your application code.

More

When you call get(), the engine follows a predictable sequence:

1) Look for an existing instance in the active scope or singleton cache.

2) Check for an explicit definition (binding) for the abstract.

3) If no definition exists, attempt autowiring using reflection and prototypes.

4) Resolve each constructor dependency using the same rules.

5) Store the instance if its lifetime requires caching, then return it.

This predictability is what makes the container reliable in production.

Methods

__construct

Initializes the container engine with its core collaborators and registers the container with the resolution engine.

More

The engine is assembled with all of its collaborators: definitions, scopes, resolution engine, prototype factory, and utilities like invokers and injectors. The goal is to keep runtime behavior centralized and predictable.

Registering the container with the resolution engine enables callbacks and advanced flows without exposing internal state publicly.

analyzeDependenciesFor

Returns a prototype builder for a given class after analysis setup.

More

This is the entry point for the Think phase. The engine delegates analysis to the prototype factory so that build plans can be created and cached.

Use this when you want to precompute or validate how the container will build a class.

design

Returns the design flow for analysis, cache, and prototype operations.

More

Design flow is a higher-level API for analyzing and optimizing service construction. It is separate from the hot path to keep runtime resolution lean.

In enterprise systems, this is typically run during bootstrap or deployment to catch issues early.

compile

Compiles and validates prototypes for all registered definitions and returns statistics.

More

Compilation is a warmup step. It forces the container to analyze and cache service prototypes so runtime resolution does not repeatedly perform reflection.

The returned stats are useful for diagnostics and performance tuning.

validate

Validates all registered definitions and their analyzed prototypes.

More

Validation confirms that all registered definitions can actually be built. This is your safety net before production boot.

It surfaces missing bindings, circular dependencies, and invalid configurations early.

clearCache

Clears cached compilation artifacts and prototype caches.

More

Clears cached prototypes and compilation metadata. Use this after changing bindings or when you want to force the container to re-analyze everything.

getCompilationStats

Returns compilation metrics from the last compile run.

More

Provides metrics from the last compilation run. This includes counts and timings that can be logged or displayed in diagnostics tools.

canInject

Checks if a target object can be injected based on reflection analysis.

More

Useful to confirm that the container understands how to inject a given object.

getInjectionInfo

Returns analyzed injection metadata for a target object.

More

Returns structured information about constructor, property, and method injection points.

get

PSR-11 entry point that resolves a service by identifier.

More

This is the main resolution method used by application code.

resolve

Resolves a service prototype into a runtime instance.

More

Uses a precomputed plan so runtime resolution is fast and predictable.

More

Checks whether the container can resolve a given identifier.

More

This does not build anything; it only answers whether the container knows about that identifier.

More

More

More

More
