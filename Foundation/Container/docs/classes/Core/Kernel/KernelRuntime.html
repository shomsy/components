KernelRuntime

Execution layer that delegates to the resolution pipeline.

Foundation/Container/Core/Kernel/KernelRuntime.php

Class Overview

KernelRuntime is the execution engine of the container kernel. It coordinates service resolution by delegating to the resolution pipeline and invocation system, maintaining clean separation between execution logic and higher-level orchestration.

More

WHY IT EXISTS:

- To provide a focused execution layer for service resolution

- To coordinate between resolution pipeline and invocation systems

- To maintain separation between execution and orchestration concerns

- To handle core resolution flow with proper error management

HOW IT FITS IN THE KERNEL:

The KernelRuntime sits at the execution core:

ContainerKernel → KernelRuntime → ResolutionPipeline

KernelRuntime → InvokeAction (for callables)

It receives resolution requests and ensures they complete successfully or fail with clear error messages.

EXECUTION MODEL:

All resolution goes through contexts that carry metadata through the pipeline. The runtime creates contexts, runs the pipeline, and validates results.

Resolution Flow Example

// Runtime executes resolution through pipeline
$runtime = new KernelRuntime($pipeline, $invoker);

// Simple resolution
$user = $runtime->get('UserService');

// With context
$context = new KernelContext(serviceId: 'UserService');
$user = $runtime->resolveContext($context);

// With overrides
$user = $runtime->make('UserService', ['id' => 123]);

// Prototype-based
$user = $runtime->resolve($prototype);
More

The runtime provides multiple resolution entry points, but all funnel through the same pipeline execution with different context configurations. This ensures consistent behavior regardless of the entry point used.

Methods

__construct

Initialize runtime with pipeline and invoker.

More

WHY IT EXISTS:

- To establish the execution dependencies

- To configure the resolution pipeline and invocation system

- To prepare the runtime for service resolution operations

DEPENDENCY INJECTION:

The runtime receives its collaborators:

ResolutionPipeline: Handles the step-by-step resolution process

InvokeAction: Manages callable execution with dependency injection

IMMUTABLE DESIGN:

The class is readonly, ensuring that runtime behavior is predictable and thread-safe once constructed.

get

Resolve a service by its identifier.

More

WHY IT EXISTS:

- To provide the primary resolution entry point

- To handle simple identifier-based resolution

- To create minimal context for straightforward requests

CONTEXT CREATION:

Creates a basic KernelContext with just the serviceId:

new KernelContext(serviceId: $id)

This context gets passed through the full pipeline without special handling.

ERROR HANDLING:

Delegates error handling to resolveContext(), which validates pipeline results and throws ResolutionException for failures.

resolveContext

Explicitly resolve with a context.

More

WHY IT EXISTS:

- To support advanced resolution scenarios with metadata

- To allow context-aware resolution decisions

- To provide detailed error information

PIPELINE EXECUTION:

Runs the resolution pipeline with the provided context:

$this->pipeline->run(context: $context)

The pipeline modifies the context, adding resolved instances, metadata, and error information.

RESULT VALIDATION:

After pipeline execution, validates that resolution succeeded:

if ($context->getInstance() === null)

Throws ResolutionException with service ID and resolution path for debugging.

CONTEXT FEATURES:

Contexts can carry:

- Service identifier and overrides

- Resolution path for debugging

- Metadata for special handling

- Instance references and error states

make

Resolve a service by its identifier with overrides.

More

WHY IT EXISTS:

- To support factory patterns with parameter overrides

- To enable contextual instance creation

- To bypass caching for fresh instances

CONTEXT WITH OVERRIDES:

Creates context with parameter overrides:

new KernelContext(serviceId: $id, overrides: $parameters)

Overrides take precedence over automatic resolution in the pipeline.

USE CASES:

- Factory methods needing specific parameters

- Test scenarios with mock dependencies

- Runtime configuration injection

resolve

Resolve a service prototype.

More

WHY IT EXISTS:

- To resolve services using pre-analyzed blueprints

- To support compilation and optimization workflows

- To bypass runtime analysis for performance

PROTOTYPE METADATA:

Attaches prototype to context for pipeline use:

$ctx->setMeta('analysis', 'prototype', $prototype)

The pipeline can use this pre-analyzed data instead of runtime reflection.

PERFORMANCE BENEFITS:

- Eliminates reflection overhead

- Provides predictable construction

- Enables ahead-of-time validation

call

Execute a callable with dependency injection.

More

WHY IT EXISTS:

- To support dependency injection for functions and methods

- To enable container-managed execution of callables

- To provide consistent parameter resolution

INVOCATION DELEGATION:

Delegates to the InvokeAction system:

$this->invoker->invoke(target: $callable, parameters: $parameters)

The invoker handles parameter analysis and dependency resolution.

CALLABLE SUPPORT:

- Global functions and closures

- Static and instance methods

- Objects with __invoke method

injectInto

Inject dependencies into an object.

More

WHY IT EXISTS:

- To inject dependencies into existing objects

- To support manual dependency injection workflows

- To enable hydration of deserialized objects

INTERNAL CONTEXT:

Creates special injection context:

new KernelContext(serviceId: '__inject__', manualInjection: true)

The special service ID signals injection-only processing to the pipeline.

INSTANCE HANDLING:

Pre-sets the target instance in context:

$ctx->resolvedWith(instance: $target)

The pipeline then injects dependencies into this existing instance.
