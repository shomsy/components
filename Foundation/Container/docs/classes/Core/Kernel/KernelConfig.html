KernelConfig

Centralized configuration for ContainerKernel

Foundation/Container/Core/Kernel/KernelConfig.php

Class Overview

KernelConfig is an immutable configuration object that holds all collaborators and settings needed by ContainerKernel. It provides a fluent API for building container configurations safely without side effects.

More

WHY IT EXISTS:

- To centralize all kernel configuration in one place

- To ensure immutable configuration changes

- To provide type-safe configuration building

- To separate configuration from kernel implementation

HOW IT FITS IN THE ARCHITECTURE:

The config acts as the glue between container components:

ContainerBuilder → KernelConfig → ContainerKernel

It contains references to all the collaborators the kernel needs to function.

IMMUTABLE DESIGN:

Every configuration change returns a new instance:

$config->withStrictMode(true) creates a new config

This ensures thread safety and predictable behavior.

FLUENT API:

Method chaining enables readable configuration:

$config->withMetrics($metrics)->withPolicy($policy)->withDevMode(false)

Application Configuration Examples

Basic Kernel Configuration

// Create basic configuration for kernel
$config = KernelConfig::create(
    engine: $resolutionEngine,
    injector: $dependencyInjector,
    invoker: $callableInvoker,
    scopes: $scopeManager,
    prototypeFactory: $serviceAnalyzer,
    timeline: $resolutionTracker
);

// Use default settings (dev mode, no strict validation)
$kernel = new ContainerKernel($definitions, $config);

Production Configuration

// Production-ready configuration
$config = KernelConfig::create(
    engine: new ProductionResolutionEngine(),
    injector: new OptimizedInjector(),
    invoker: new CachedInvoker(),
    scopes: new ScopedManager(),
    prototypeFactory: new CompiledPrototypeFactory(),
    timeline: new MetricsTimeline()
)->withStrictMode(true)       // Enable strict validation
 ->withDevMode(false)         // Disable dev features
 ->withMetrics($metrics)      // Add performance monitoring
 ->withPolicy($securityPolicy); // Add security checks

$kernel = new ContainerKernel($definitions, $config);

Development Configuration

// Development configuration with extra debugging
$config = KernelConfig::create(
    engine: new DebugResolutionEngine(),
    injector: new VerboseInjector(),
    invoker: new LoggingInvoker(),
    scopes: new ScopeManager(),
    prototypeFactory: new DebugPrototypeFactory(),
    timeline: new DetailedTimeline()
)->withDevMode(true)          // Enable dev features
 ->withAutoDefine(true)       // Auto-register services
 ->withMetrics($debugMetrics) // Detailed metrics
 ->withTerminator($cleanupHandler); // Cleanup on shutdown

$kernel = new ContainerKernel($definitions, $config);

Custom Builder Pattern

// Custom configuration builder
class ContainerConfigurator {
    private KernelConfig $config;

    public function __construct(KernelConfig $baseConfig) {
        $this->config = $baseConfig;
    }

    public function forProduction(): self {
        $this->config = $this->config
            ->withStrictMode(true)
            ->withDevMode(false)
            ->withMetrics(new ProductionMetrics());
        return $this;
    }

    public function forTesting(): self {
        $this->config = $this->config
            ->withStrictMode(false)
            ->withDevMode(true)
            ->withAutoDefine(true);
        return $this;
    }

    public function withSecurity(ContainerPolicy $policy): self {
        $this->config = $this->config->withPolicy($policy);
        return $this;
    }

    public function build(): KernelConfig {
        return $this->config;
    }
}

Environment-Based Configuration

// Environment-aware configuration
class EnvironmentConfig {
    public static function create(string $environment): KernelConfig {
        $baseConfig = KernelConfig::create(/* ... required params ... */);

        return match ($environment) {
            'production' => $baseConfig
                ->withStrictMode(true)
                ->withDevMode(false)
                ->withMetrics(new CloudWatchMetrics()),

            'staging' => $baseConfig
                ->withStrictMode(true)
                ->withDevMode(true)
                ->withMetrics(new LogMetrics()),

            'development' => $baseConfig
                ->withStrictMode(false)
                ->withDevMode(true)
                ->withAutoDefine(true)
                ->withMetrics(new DebugMetrics()),

            'testing' => $baseConfig
                ->withStrictMode(false)
                ->withDevMode(false)
                ->withAutoDefine(true),
        };
    }
}

Methods

__construct

Initialize kernel configuration with all collaborators.

More

WHY IT EXISTS:

- To accept all kernel collaborators in a type-safe way

- To establish the complete configuration state

- To validate configuration requirements at creation time

REQUIRED COLLABORATORS:

These are essential for kernel operation:

EngineInterface: Handles service resolution logic

InjectDependencies: Manages dependency injection

InvokeAction: Handles callable execution

ScopeManager: Manages instance lifetimes

ServicePrototypeFactory: Creates service analysis blueprints

ResolutionTimeline: Tracks resolution performance

OPTIONAL COLLABORATORS:

These enhance functionality when provided:

CollectMetrics: Performance monitoring

ContainerPolicy: Security and access control

TerminateContainer: Graceful shutdown handling

CONFIGURATION FLAGS:

autoDefine: Automatically register concrete classes

strictMode: Enable strict validation and error checking

devMode: Enable development-time features and debugging

create

Create configuration with required collaborators only.

More

WHY IT EXISTS:

- To provide a simple factory method for basic configurations

- To reduce boilerplate when optional features aren't needed

- To ensure required collaborators are always provided

DIFFERENCE FROM __construct:

create() only requires essential collaborators:

- No metrics, policy, or terminator (set to null)

- Default flags: autoDefine=false, strictMode=false, devMode=true

WHEN TO USE:

- Simple applications without advanced features

- Testing scenarios with minimal setup

- Starting point for further configuration with fluent methods

BUILDING ON CREATE:

Use fluent methods to add optional features:

KernelConfig::create(...)->withMetrics($m)->withPolicy($p)

withStrictMode

Enable or disable strict validation mode.

More

WHY IT EXISTS:

- To control validation strictness based on environment

- To catch configuration errors early in strict mode

- To allow lenient operation during development

STRICT MODE BEHAVIOR:

When enabled:

- Throws exceptions on ambiguous bindings

- Validates all injection points strictly

- Requires explicit service registrations

- Performs comprehensive error checking

WHEN TO ENABLE:

- Production environments

- CI/CD pipelines

- When you want to catch errors early

WHEN TO DISABLE:

- Development with rapid prototyping

- Legacy code integration

- When flexibility is more important than strictness

withAutoDefine

Enable or disable automatic service definition.

More

WHY IT EXISTS:

- To reduce boilerplate in simple applications

- To automatically register concrete classes as services

- To support convention-over-configuration patterns

AUTO-DEFINE BEHAVIOR:

When enabled:

- Concrete classes can be resolved without explicit binding

- Constructor injection works automatically

- Reduces need for manual service registration

PERFORMANCE IMPACT:

- Slight overhead for reflection-based resolution

- May create services that aren't actually needed

- Best for development; consider explicit bindings for production

WHEN TO USE:

- Small applications with straightforward dependencies

- Development and prototyping

- When you want minimal configuration

withMetrics

Configure metrics collection.

More

WHY IT EXISTS:

- To enable performance monitoring and observability

- To track container usage patterns

- To support debugging and optimization

METRICS CAPTURED:

- Resolution counts and timing

- Cache hit/miss ratios

- Scope usage statistics

- Error rates and types

IMPLEMENTATION:

The metrics collector receives events from:

- Resolution engine (successful/failed resolutions)

- Scope manager (instance creation/destruction)

- Compiler (validation and caching operations)

USE CASES:

- Production monitoring and alerting

- Performance profiling and optimization

- Debugging container usage patterns

withPolicy

Configure security policy.

More

WHY IT EXISTS:

- To enforce security constraints on service resolution

- To control which services can be accessed

- To implement organizational security policies

POLICY CONTROLS:

- Service access restrictions

- Namespace-based whitelisting/blacklisting

- Role-based access control

- Custom security rules

WHEN THROWS EXCEPTIONS:

- Unauthorized service access attempts

- Policy violations during resolution

- Forbidden constructor parameters

USE CASES:

- Multi-tenant applications

- Enterprise security requirements

- Regulatory compliance

withTerminator

Configure shutdown terminator.

More

WHY IT EXISTS:

- To enable graceful shutdown of container services

- To clean up resources and connections

- To handle application termination properly

TERMINATION PROCESS:

- Called during container shutdown

- Disposes scoped instances

- Closes connections and releases resources

- Calls cleanup methods on services

LIFECYCLE INTEGRATION:

Works with:

- PHP shutdown handlers

- Framework termination events

- Application cleanup routines

USE CASES:

- Long-running applications (daemons, servers)

- Proper resource cleanup

- Database connection management

withDevMode

Enable or disable development mode.

More

WHY IT EXISTS:

- To enable development-time features and debugging

- To provide better error messages during development

- To disable performance optimizations that aid debugging

DEV MODE FEATURES:

When enabled:

- More detailed error messages and stack traces

- Additional validation and warnings

- Debug logging and diagnostics

- Relaxed security checks for easier development

PERFORMANCE IMPACT:

- Slightly slower due to extra validation

- More memory usage for debug information

- Should always be disabled in production

WHEN TO ENABLE:

- Local development environments

- Staging environments for testing

- Any non-production deployment
