KernelStep

Contract for resolution pipeline steps

Foundation/Container/Core/Kernel/Contracts/KernelStep.php

Interface Overview

KernelStep defines the contract that all resolution pipeline steps must implement. Pipeline steps are the individual processing units that collaborate to resolve a service, each handling a specific aspect of the resolution process.

More

WHY IT EXISTS:

- To establish a standard interface for pipeline components

- To enable composable resolution logic

- To support different step implementations for various concerns

- To allow pipeline steps to be tested and replaced independently

HOW IT FITS IN THE PIPELINE:

The pipeline executes steps in sequence:

Pipeline → Step1 → Step2 → Step3 → Resolution

Each step receives the same KernelContext and can read/modify it before passing to the next step.

STEP TYPES:

- Binding resolution steps

- Scope checking steps

- Autowiring steps

- Injection steps

- Validation steps

Pipeline Execution Example

// Pipeline executes steps in sequence
$pipeline = new ResolutionPipeline([
    new ScopeCheckStep(),
    new BindingResolutionStep(),
    new AutowireStep(),
    new DependencyInjectionStep()
]);

$context = new KernelContext(serviceId: 'UserService');
$pipeline->run($context);

// Each step processes the context
final class ScopeCheckStep implements KernelStep {
    public function __invoke(KernelContext $context): void {
        // Check if service is cached in active scopes
        if ($this->scopes->has($context->serviceId)) {
            $context->resolvedWith($this->scopes->get($context->serviceId));
        }
    }
}

final class BindingResolutionStep implements KernelStep {
    public function __invoke(KernelContext $context): void {
        // Only run if not already resolved
        if (!$context->isResolved()) {
            $binding = $this->findBinding($context->serviceId);
            if ($binding) {
                $context->setMeta('binding', 'resolved', $binding);
            }
        }
    }
}
More

The pipeline runs each step in order, allowing early steps to "short-circuit" resolution by setting the instance. Later steps can check if resolution is already complete and skip their processing.

Implementing a Custom Step

// Custom step for logging resolution
final class LoggingStep implements KernelStep {
    public function __construct(
        private Logger $logger
    ) {}

    public function __invoke(KernelContext $context): void {
        $start = microtime(true);

        // Log the resolution attempt
        $this->logger->debug('Resolving service: ' . $context->serviceId);

        // Continue with next steps...

        // Could log completion here if needed
        // But typically logging is done by telemetry
    }
}

// Step that adds custom metadata
final class CustomMetadataStep implements KernelStep {
    public function __invoke(KernelContext $context): void {
        // Add custom information to context
        $context->setMeta('custom', 'timestamp', time());
        $context->setMeta('custom', 'environment', getenv('APP_ENV'));

        // This metadata can be used by later steps
        // or accessed after resolution for debugging
    }
}

// Validation step
final class ValidationStep implements KernelStep {
    public function __invoke(KernelContext $context): void {
        if (!$context->isResolved()) {
            // Service wasn't resolved by previous steps
            if (!$context->allowAutowire) {
                throw new ResolutionException(
                    "Service {$context->serviceId} not found and autowiring disabled"
                );
            }
        }
    }
}
More

Custom steps can be added to the pipeline for specific application needs. Common use cases include logging, validation, caching, security checks, and custom resolution logic.

Method

__invoke

Execute this step in the resolution pipeline.

More

WHY IT EXISTS:

- To define the standard execution interface for pipeline steps

- To allow steps to be called as functions for clean syntax

- To enable dependency injection of the context

EXECUTION CONTEXT:

The method receives a KernelContext containing:

- Service identifier and resolution state

- Configuration flags (debug, autowire, etc.)

- Metadata from previous steps

- Instance if already resolved

MODIFICATION RIGHTS:

Steps can modify the context by:

- Setting the resolved instance

- Adding metadata

- Setting flags or overrides

- Triggering errors

RESPONSIBILITY:

Each step should:

1. Check if resolution is already complete

2. Perform its specific processing

3. Update context with results or metadata

4. Allow pipeline to continue or short-circuit

ERROR HANDLING:

Steps can throw exceptions to halt resolution:

- ResolutionException for resolution failures

- LogicException for programming errors

- Custom exceptions for specific validation failures

PERFORMANCE CONSIDERATIONS:

- Keep processing lightweight

- Avoid expensive operations when possible

- Use early returns when resolution is complete

- Cache expensive computations in context metadata

Step Categories

Early Steps (Short-circuit): Check caches, scopes, bindings. Can complete resolution early.

Middle Steps (Processing): Perform analysis, validation, transformation. Add metadata.

Late Steps (Finalization): Handle injection, decoration, cleanup. Finalize resolution.

Error Steps (Recovery): Handle failures, provide fallbacks, log errors.
