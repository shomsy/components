TerminalKernelStep

Marker for steps that can short-circuit resolution

Foundation/Container/Core/Kernel/Contracts/TerminalKernelStep.php

Interface Overview

TerminalKernelStep is a marker interface that extends KernelStep. It indicates that implementing steps have the potential to terminate the resolution pipeline early by providing a complete resolution result.

More

WHY IT EXISTS:

- To identify steps that can short-circuit expensive processing

- To optimize resolution performance by avoiding unnecessary work

- To provide semantic information about step capabilities

- To enable pipeline optimizations and step reordering

HOW IT WORKS:

Terminal steps can complete resolution by:

1. Finding a cached instance in scope

2. Locating an existing singleton

3. Providing a pre-resolved result

4. Short-circuiting further pipeline processing

PIPELINE OPTIMIZATION:

The pipeline can use this marker to:

- Run terminal steps first for better performance

- Skip remaining steps when resolution is complete

- Optimize step execution order

Terminal Step Examples

Scope Check Step

// Checks if service exists in current scope
final class ScopeCheckStep implements TerminalKernelStep {
    public function __invoke(KernelContext $context): void {
        $scopes = $this->scopeManager->getActiveScopes();

        foreach ($scopes as $scope) {
            if ($scope->has($context->serviceId)) {
                // Found in scope - terminate pipeline
                $instance = $scope->get($context->serviceId);
                $context->resolvedWith($instance);
                return; // Pipeline stops here
            }
        }

        // Not found - continue to next step
    }
}
More

WHY TERMINAL: If an instance exists in any active scope, no further resolution is needed. The pipeline can immediately return the cached instance.

PERFORMANCE IMPACT: O(1) scope lookups can save milliseconds of reflection and construction time.

Singleton Cache Step

// Checks singleton registry for cached instances
final class SingletonCacheStep implements TerminalKernelStep {
    private array $singletons = [];

    public function __invoke(KernelContext $context): void {
        if (isset($this->singletons[$context->serviceId])) {
            // Singleton exists - terminate pipeline
            $context->resolvedWith($this->singletons[$context->serviceId]);
            return;
        }

        // Check if this should be a singleton
        $definition = $this->definitions->get($context->serviceId);
        if ($definition && $definition->lifetime === ServiceLifetime::Singleton) {
            // First resolution of singleton - create and cache
            // (Actual resolution happens in later steps)
        }
    }
}
More

WHY TERMINAL: Singletons are globally cached. Once resolved, they should be returned immediately on subsequent requests.

CACHE STRATEGY: Combines existence checks with creation hints for optimal singleton handling.

Early Binding Resolution

// Resolves explicit bindings before complex logic
final class BindingResolutionStep implements TerminalKernelStep {
    public function __invoke(KernelContext $context): void {
        $binding = $this->definitions->findBinding($context->serviceId);

        if ($binding && $binding->isResolvedInstance()) {
            // Pre-resolved instance binding
            $context->resolvedWith($binding->getInstance());
            return;
        }

        if ($binding && $binding->isFactory()) {
            // Factory binding - execute and terminate
            $instance = $binding->executeFactory($context);
            $context->resolvedWith($instance);
            return;
        }

        // No binding or complex binding - continue pipeline
    }
}
More

WHY TERMINAL: Simple bindings (instances, factories) can be resolved immediately without further processing.

COMPLEXITY HANDLING: Only terminates for simple cases; complex bindings continue through full pipeline.

Pipeline Flow with Terminal Steps

// Pipeline execution with terminal optimization
class ResolutionPipeline {
    private array $steps;

    public function run(KernelContext $context): void {
        foreach ($this->steps as $step) {
            $step($context);

            // If terminal step resolved, stop processing
            if ($context->isResolved()) {
                break;
            }
        }
    }
}

// Example flow:
// 1. ScopeCheckStep (terminal) - checks cache → FOUND → STOP
// 2. BindingStep (terminal) - not reached
// 3. AutowireStep - not reached
// 4. InjectStep - not reached

// Alternative flow:
// 1. ScopeCheckStep (terminal) - checks cache → NOT FOUND → CONTINUE
// 2. BindingStep (terminal) - finds factory → EXECUTE → RESOLVED → STOP
// 3. AutowireStep - not reached
// 4. InjectStep - not reached
More

SHORT-CIRCUIT OPTIMIZATION: Terminal steps enable the pipeline to stop as soon as resolution is complete, avoiding unnecessary work.

ORDER MATTERS: Fast terminal steps (like scope checks) should run first for maximum performance benefit.

Implementation Guidelines

When to Implement TerminalKernelStep

Fast Existence Checks: If you can quickly determine if resolution is possible without full processing.

Cached Results: When you have pre-computed or cached instances available.

Simple Bindings: For straightforward mappings that don't need complex resolution logic.

Optimization Opportunities: When early termination can significantly improve performance.

When NOT to Use Terminal Steps

Complex Logic: Don't terminate if the step needs to contribute metadata for later steps.

Partial Results: Avoid if the step provides incomplete resolution that needs further processing.

Side Effects: Don't terminate if later steps are needed for proper initialization or injection.

Validation: Regular KernelStep is better if the step performs important validation or setup.

Marker Interface Benefits

Design Clarity: Explicitly marks steps with short-circuit capability.

Pipeline Optimization: Enables reordering and optimization of step execution.

Testing: Allows targeted testing of terminal behavior.

Documentation: Makes step capabilities clear to maintainers.

Relationship to KernelStep

Inheritance: TerminalKernelStep extends KernelStep, so all terminal steps are also regular steps.

Additional Contract: Terminal steps promise they may complete resolution, but are not required to.

Pipeline Compatibility: Terminal steps work in any pipeline position but are most effective early.

Observability: Terminal steps are subject to the same telemetry and monitoring as regular steps.
