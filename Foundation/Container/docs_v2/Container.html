Container

The facade you call to resolve services while the kernel and pipelines do the heavy lifting.

File Path: Container.php
Folder Context: Root

Quick Summary

The Container class is the public API for dependency resolution. It delegates to ContainerKernel for resolution, scoping, and lifecycle management while keeping a small surface of methods like get(), make(), and call(). It exists so application code never touches the lower-level kernel or pipelines directly.

For Humans: What This Means
You ask this class for anything you need. Its the front desk that forwards your request to the back office and hands you a ready-to-use service.

Terminology

Dependency Injection Container
System that builds objects and supplies their dependencies automatically.
Facade
Thin API hiding complex internalsin this case, the kernel and pipelines.
Resolution
Turning a service identifier into a fully constructed object.
Scope
Lifetime boundary (singleton, scoped, transient) for resolved services.
Prototype
Precomputed recipe describing how to build a service.
Injection Report
Metadata describing which properties/methods can be injected.
For Humans: What This Means
These are the parts behind the counter. The container knows how long to keep things, how to assemble them, and where to inject them. You only interact with the front door.

Think of It

Picture a hotel: the front desk (Container) takes your request, the back office (ContainerKernel) finds a room, housekeeping (scopes) manages reuse, and engineering (pipelines) keeps everything running.

HotelIn Code
Front deskContainer facade methods
Back office routingContainerKernel resolution pipeline
Room assignmentScope manager lifetimes
HousekeepingInjection checks and reuse
For Humans: What This Means
You dont go to the boiler room; you talk to the desk. The desk coordinates and hands you a key. Thats what $container->get() does.

Story Example

Before: Controllers manually new-up services, duplicate wiring, and leak configuration everywhere.

After: Bootstrap registers bindings once; app code calls $container->get(OrderProcessor::class). The container builds the graph, applies scopes, injects dependencies, and returns a ready instance.

For Humans: What This Means
You stop scattering constructors and credentials. You ask for what you need; the container assembles and returns it consistently.

For Dummies

Problem solved: Manual wiring is repetitive and error-prone.

How it works: Register rules, then call get() or make(). The container checks scopes, bindings, autowires, injects, and returns.

Common misconceptions: Containers are slow (they cache and scope), Just use new everywhere (you lose testability), Make everything singleton (state leaks across requests).

FAQ: Do I call the kernel directly? No, use the facade. Do I register every class? No, concretes can autowire.

How It Works (Technical)

  1. Facade entry: Methods delegate to ContainerKernel.
  2. Resolution pipeline: Scope lookup binding lookup autowiring injection lifecycle storage.
  3. Scopes: ScopeManager decides reuse vs new instances.
  4. Injection: InjectionReport guides property/method injection.
  5. Metrics: Telemetry/timelines can emit resolution diagnostics.
For Humans: What This Means
Every request follows the same conveyor belt: check cache, build if needed, wire dependencies, keep or discard per lifetime, and log what happened.

Architecture Role

Acts as the composition root interface. It isolates controllers/services from kernel internals and preserves backward compatibility when kernel details evolve.

For Humans: What This Means
Your app relies on this one doorway. Internals can change without breaking callers, keeping your code stable.

Risks, Trade-offs & Recommended Practices

Risk: Grabbing the container deep in business logic hides dependencies.
Recommended: Prefer constructor injection; keep container use at bootstrap.

Risk: Mis-scoped services leak state across requests.
Recommended: Use scoped lifetimes intentionally; singleton only for stateless services.

Risk: Overbinding singletons can mask resource leaks.
Recommended: Choose lifetimes deliberately and dispose where needed.

For Humans: What This Means
Be explicit about lifetimes and avoid sprinkling container calls everywhere. That keeps state safe and code testable.

Related Files & Folders

  • Core/ContainerKernel.php: Resolution pipeline and lifecycle orchestration.
  • Core/ContainerConfig.php: Kernel configuration (bindings, prototype factory, metrics).
  • Features/Define/Store/*: Stores service definitions used during resolution.
  • Features/Operate/Scope/*: Manages scopes and lifetimes.
  • Features/Think/*: Prototype analysis and caching for autowiring.
For Humans: What This Means
The facade talks to these departments: kernel dispatch, definition storage, scope management, and prototype analysis. Together they make ask and receive possible.

Methods (Anchors for @see)

  • get(string $id): mixed resolve a service identifier.
  • has(string $id): bool check resolvability without constructing.
  • make(string $abstract, array $parameters = []): object force a new instance.
  • call(callable|string $callable, array $parameters = []): mixed invoke with DI.
  • injectInto(object $target): object inject into an existing object.
  • resolve(ServicePrototype $prototype): mixed resolve from a prototype.
  • resolveContext(KernelContext $context): mixed resolve with context.
  • instance(string $abstract, object $instance): void register a shared instance.
  • beginScope(): void start a new scope.
  • endScope(): void end the current scope.
  • canInject(object $target): bool check injection viability.
  • inspectInjection(object $target): InjectionReport inspect injection points.
  • scopes(): ScopeManager access scope manager.
  • exportMetrics(): string export usage/metrics.
  • getDefinitions(): DefinitionStore access definition storage.
